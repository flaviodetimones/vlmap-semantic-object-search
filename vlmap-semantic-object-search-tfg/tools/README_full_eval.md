# Full 2x2 Evaluation Harness

This repository supports a full online `2x2` evaluation matrix with two
orthogonal axes:

- Orchestrator: `baseline` or `executor`
- Heatmap: `baseline` or `postprocessed`

The four evaluated methods are:

- `Ob_Hb`
- `Ob_Hp`
- `Oe_Hb`
- `Oe_Hp`

The intended workflow is to run this same `2x2` matrix multiple times with
different YOLOE confidence thresholds, typically:

- `0.30`
- `0.35`
- `0.40`

This keeps the method comparison fixed while isolating the effect of the
detector confidence threshold.

## Main scripts

- `tools/run_full_eval.py`
  Runs the full `2x2` pipeline for one or more scenes, supports `--methods`
  for subsets, and supports `--resume` / `--force` for long runs.
- `tools/aggregate_full_eval.py`
  Aggregates per-query and per-method results from the manifests generated by
  `run_nav_eval.py`.
- `tools/plot_full_eval.py`
  Produces compact PNG summaries from the cross-scene aggregate CSV.
- `tools/run_nav_eval.py`
  Runs one method on one scene / one JSONL and writes `manifest.json`,
  `setup.log`, `raw_log.txt` and query segments.

Compatibility shims are kept:

- `tools/run_full_2x2_eval.py`
- `tools/aggregate_full_2x2_eval.py`

## Primary metrics

The harness computes more than eight metrics, but the primary reporting set is:

- `SR`
- `Object SR`
- `CFR`
- `CT2R`
- `Rooms Before Success`
- `Wrong Visits`
- `Mean Pose Updates`
- `Early Stop Rate`

These are the metrics used in:

- the main per-scene markdown table
- the cross-scene markdown table
- the default bar plots
- the summary heatmap

Secondary metrics remain available in CSV / markdown for diagnosis:

- `Room SR`
- `Inspections Before Success`
- `Final Room Accuracy`
- `Preview No Action Rate`
- `Path Stopped Early Rate`
- `Mean Executor Actions`
- `Mean Room Transitions`

## YOLOE threshold sweep

`run_nav_eval.py` accepts:

```bash
--yoloe-conf-thresh 0.30
```

and stores that value in:

- `setup.log`
- `manifest.json`
- root `config.json` of the run

The recommended experiment pattern is:

1. Run the full `2x2` with `--yoloe-conf-thresh 0.30`
2. Run the same `2x2` again with `0.35`
3. Run the same `2x2` again with `0.40`
4. Compare the three result roots separately

## Resume semantics

When `--resume` is passed, `run_full_eval.py` skips any `(scene, method)` pair
whose `manifest.json` already exists. This allows long runs to be resumed after
interruption without repeating completed cells.

Use `--force` to rerun everything even if manifests already exist.

## Reporting

At the end of a run the harness writes:

- `aggregate_metrics.md`
  Per-scene summary with the primary table, secondary metrics and contrast
  blocks for:
  - action vocabulary
  - heatmap postprocessing
- `aggregate_cross_scenes.md`
  Cross-scene `mean ± std` table for the primary metrics
- `report.html`
  Self-contained static report that embeds the cross-scene table, plots and
  links / tables for each per-scene output directory

## Typical commands

Run the full matrix on one scene with threshold `0.30`:

```bash
python tools/run_full_eval.py \
  --scene-ids 0 \
  --dataset-type hssd \
  --data-paths hssd \
  --scene-dataset-config-file /workspace/data/versioned_data/hssd-hab/hssd-hab.scene_dataset_config.json \
  --yoloe-conf-thresh 0.30 \
  --out /workspace/results/eval_runs/demo_thr030
```

Run only two methods for a smoke test:

```bash
python tools/run_full_eval.py \
  --scene-ids 0 \
  --methods Ob_Hb,Oe_Hb \
  --dataset-type hssd \
  --data-paths hssd \
  --scene-dataset-config-file /workspace/data/versioned_data/hssd-hab/hssd-hab.scene_dataset_config.json \
  --yoloe-conf-thresh 0.30 \
  --out /workspace/results/eval_runs/smoke_thr030
```

Resume a partial long run:

```bash
python tools/run_full_eval.py \
  --scenes-from-config /workspace/results/eval_runs/20260423_120000/config.json \
  --resume \
  --out /workspace/results/eval_runs/20260423_120000
```

## YOLO conf-thresh sweep

When you only want to study the effect of the YOLOE verification threshold, use
the dedicated sweep harness instead of rerunning the whole `2x2`.

Example:

```bash
python tools/run_yoloe_thresh_sweep.py \
    --scene-id 0 \
    --queries tools/eval_queries/102344193_0.jsonl \
    --thresholds 0.30,0.40,0.50,0.60 \
    --out /workspace/results/yoloe_sweep/$(date +%Y%m%d_%H%M%S)
```

By default the sweep runs a single method, `Oe_Hp`, because the threshold is
orthogonal to the orchestrator / heatmap axis. That keeps the cost low while
still exposing the precision / recall tradeoff of YOLOE verification.
